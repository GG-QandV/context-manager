# Архитектурная схема подключения внешних чатов через SSH-туннель

Данный документ описывает единую структуру взаимодействия всех компонентов интеграции, включая сетевые потоки (Data Flow) и логику управления процессами (Control Flow) в контексте Windows 10/11 и Linux.

---

## Расшифровка потоков и элементов схемы

### 1. Потоки данных (Data Flow — Сетевой трафик MCP)
* **`[1]` -> `[2]`**: Внешний AI-клиент отправляет зашифрованный HTTPS-запрос на уникальный адрес `serveo.net`, который пробрасывает его внутрь защищенного SSH-соединения на локальный порт шлюза `:8769`.
* **`[3]`**: Шлюз **[Ж] `mcp_oauth_adapter.py`** принимает запрос, валидирует Bearer-токен (сгенерированный в **`common.py`**) и направляет его:
  - На порт `:8765` (**[З] `mcp_sse_adapter.py`**) для SSE-клиентов (Claude, Grok).
  - На порт `:8770` (**[И] Node.js `cm_http_adapter.mjs`**) для HTTP-клиентов (Perplexity, ChatGPT).
* **`[4]`**: Внутренние адаптеры разбирают JSON-RPC и выполняют локальные запросы к ядру Fastify API **[К]** на порт `:3847` для вызова требуемых инструментов context-manager, работающих с СУБД и векторной базой **[Л]**.

### 2. Жизненный цикл и управление процессами (Control Flow)
* **`[А]`**: Системный трей **`tray_pyqt.py`** (применяющий метаданные **`ui_meta.py`** для рендеринга иконок меню) позволяет пользователю запустить или остановить туннелирование.
* **`[Б]`**: Трей запускает фоновый процесс оркестратора **`tunnel_manager.py`** (**`[В]`**).
* **`[Г]`**: Оркестратор обращается к **`tunnel_providers.py`** для сборки аргументов SSH, проверяет PID-файлы и порты через **`tunnel_state.py`** и стартует процесс SSH-клиента **`[Е]`**.
* **`[Д]`**: После успешного получения публичного URL от туннеля, оркестратор запускает процесс шлюза **`mcp_oauth_adapter.py`** (**`[Ж]`**).
* **`common.py`** обеспечивает все процессы едиными путями сохранения настроек и токенами доступа.
