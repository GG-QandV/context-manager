# Отчет о совместимости модулей Context Manager с Windows 10/11

Данный документ содержит детальный анализ совместимости всех перенесенных и интегрированных модулей, скриптов установки и компонентов автозапуска в проекте `context-manager` для операционных систем Windows 10 и Windows 11.

---

## 1. Системный трей и UI (`tray_pyqt.py`)

* **Технологический стек**: `PyQt6` (а именно `QtWidgets`, `QtGui`, `QtCore`).
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - **Графический интерфейс**: PyQt6 компилирует нативные виджеты Windows и отлично интегрируется в системный трей (System Tray) Windows 10/11.
  - **Иконки**: Замена эмодзи на SVG-файлы выполнена через кроссплатформенный класс `QIcon`. В отличие от эмодзи, которые могут рендериться некорректно или различаться в зависимости от шрифтов Windows, векторные SVG-иконки отображаются идентично на всех платформах.
  - **Фоновый запуск**: Для предотвращения открытия пустого черного окна консоли `cmd.exe` при старте трея и запускаемых им скриптов, используется запуск через интерпретатор `pythonw.exe`. Скрипт автоматически ищет `pythonw.exe` в том же виртуальном окружении, где запущен `python.exe`.
  - **Длина Tooltip**: В Windows длина всплывающей подсказки трея (`setToolTip`) ограничена 127 символами. В коде реализована защита:
    ```python
    MAX_TOOLTIP_WIN = 127
    if sys.platform == "win32" and len(tooltip) > MAX_TOOLTIP_WIN:
        tooltip = tooltip[:MAX_TOOLTIP_WIN - 1] + "…"
    ```

## 2. Управление процессами и портами (`tunnel_state.py`)

* **Технологический стек**: `psutil`, `subprocess`, системные утилиты Windows.
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - **Проверка PID**: Используется библиотека `psutil.pid_exists(pid)`, которая является кроссплатформенным и наиболее стабильным стандартом для Windows.
  - **Убийство зависших процессов**: При отсутствии библиотеки `psutil` реализован надежный Windows-fallback с использованием системной утилиты `taskkill`:
    ```python
    if sys.platform == "win32":
        subprocess.run(["taskkill", "/F", "/PID", str(snap.pid)], capture_output=True)
    ```
  - **Очистка занятых портов**: Windows не поддерживает утилиты `lsof` и `fuser`. Для поиска и принудительного завершения процессов, занимающих порт (например, OAuth-адаптер на порту `8769`), на Windows вызывается парсинг вывода `netstat -ano`:
    ```python
    result = subprocess.run(["netstat", "-ano"], capture_output=True, text=True, timeout=5)
    # ... парсинг строки и получение PID ...
    subprocess.run(["taskkill", "/F", "/PID", str(pid)], capture_output=True)
    ```
    Если в окружении установлен `psutil`, очистка портов происходит еще быстрее и надежнее через `psutil.net_connections(kind="tcp")`.

## 3. Жизненный цикл туннеля (`tunnel_manager.py`)

* **Технологический стек**: `asyncio`, `subprocess`, `signal`.
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - **Определение headless-режима**: На Windows headless-режим определяется по переменной окружения `SESSIONNAME` (которая пуста, если сессия не является интерактивной):
    ```python
    if sys.platform == "win32":
        return os.environ.get("SESSIONNAME", "") == ""
    ```
  - **Обработка сигналов завершения**: В Windows системный цикл асинхронных событий `asyncio` не поддерживает методы добавления обработчиков сигналов `loop.add_signal_handler` (вызывает `NotImplementedError`). В коде это учтено и безопасно обернуто:
    ```python
    for sig in (signal.SIGINT, signal.SIGTERM):
        try:
            loop.add_signal_handler(sig, stop_event.set)
        except NotImplementedError:
            pass  # Пропускается на Windows
    ```
    Завершение на Windows корректно отрабатывает через стандартное исключение `KeyboardInterrupt`.

## 4. Провайдеры туннеля (`tunnel_providers.py`)

* **Технологический стек**: `ssh`, `autossh`, `shutil`, `shlex`, `subprocess`.
* **Совместимость**: **Полная** (при наличии OpenSSH клиента в Windows).
* **Анализ деталей**:
  - **Наличие SSH**: Скрипт ищет исполняемый файл `ssh` или `autossh` в системе с помощью `shutil.which("ssh")`. В Windows 10 (начиная с сборки 1809) и Windows 11 клиент OpenSSH поставляется предустановленным по умолчанию.
  - **Парсинг аргументов**: Для правильного разбиения командной строки на Windows используется флаг `posix=False` в `shlex.split`:
    ```python
    def _build_cmd_args(cmd: str) -> list[str]:
        return shlex.split(cmd, posix=(sys.platform != "win32"))
    ```
  - **Группы процессов**: При запуске `ssh` на Windows процесс создается в новой группе процессов:
    ```python
    if sys.platform == "win32":
        kwargs["creationflags"] = subprocess.CREATE_NEW_PROCESS_GROUP
    ```
    Это позволяет корректно отправлять сигнал Ctrl+C дочернему процессу Serveo (`self._proc.send_signal(signal.CTRL_C_EVENT)`) при остановке туннеля, не прерывая работу самого Python-скрипта трея.

## 5. Общий модуль и пути конфигурации (`common.py`)

* **Технологический стек**: `os`, `pathlib`, `sys`.
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - **Директории конфигурации**: Путь к файлам настроек iFlow и токенов на Windows корректно разрешается в папку `%APPDATA%` (обычно `C:\Users\<Имя>\AppData\Roaming\iflow`):
    ```python
    elif sys.platform == "win32":
        appdata = os.environ.get("APPDATA")
        if appdata:
            return Path(appdata) / "iflow"
        return Path.home() / "AppData" / "Roaming" / "iflow"
    ```
  - **Атомарная запись токенов**: Запись токенов авторизации происходит через создание временного файла и его замену (`os.replace`). Приватные права доступа `chmod(0o600)` на Windows пропускаются, так как Windows использует списки контроля доступа (ACL) вместо битовых масок POSIX.

## 6. Адаптеры MCP SSE и OAuth (`mcp_sse_adapter.py`, `mcp_oauth_adapter.py`)

* **Технологический стек**: `starlette`, `uvicorn`, `httpx`, `mcp`.
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - Веб-серверы написаны на чистом Python с использованием асинхронных библиотек. Они не привязаны к специфичным механизмам Unix (таким как Unix Domain Sockets) и работают на стандартных TCP/IP сокетах (порты `8765`, `8769`).

---

## 7. Windows-инсталлятор (`context-manager-setup.iss`)

* **Технологический стек**: Inno Setup, PowerShell, `winget`.
* **Совместимость**: **Полная** (Windows 10 версии 1809+ / Windows 11).
* **Анализ деталей**:
  - **Автоматизация**: Инсталлятор скачивает и устанавливает зависимости (Node.js LTS, PostgreSQL 16) через системный пакетный менеджер `winget.exe`.
  - **Службы Windows**: Фоновые процессы (`cm-api`, `cm-embed`, `cm-mcp`, `cm-watchdog`) регистрируются как нативные Windows-службы с помощью утилиты `nssm.exe` (Non-Sucking Service Manager). Это гарантирует их автозапуск при старте системы без необходимости входа пользователя в систему.
  - **Автозапуск трея**: Ярлык запуска системного трея создается в пользовательской папке автозагрузки `{userstartup}` и указывает на `pythonw.exe` для скрытого фонового режима.

---

## 8. PowerShell установщик (`scripts/install.ps1`)

* **Технологический стек**: PowerShell 5.1+ / PowerShell Core.
* **Совместимость**: **Полная**.
* **Анализ деталей**:
  - Скрипт автоматически создает виртуальное окружение Python (`.venv`), скачивает зависимости через локальный `pip` и регистрирует ярлык автозапуска трея в Startup-папке пользователя с помощью COM-объекта `WScript.Shell`.

---

## Резюме

Все программные модули, утилиты интеграции туннеля и установщики **полностью совместимы** с операционными системами Windows 10 и Windows 11. В коде учтены все основные платформенные различия Windows:
1. Замена UNIX-сигналов на Windows `taskkill` и `CTRL_C_EVENT`.
2. Поддержка интерпретатора `pythonw.exe` для скрытия пустых консольных окон.
3. Разрешение путей через `%APPDATA%`.
4. Парсинг вывода `netstat -ano` вместо отсутствующего в Windows `lsof`.
5. Использование Inno Setup и `nssm` для развертывания фоновых служб Windows.
