# Аудит багов и "затыков" в WIN10_11_INSTALL_WORKFLOW.md

Данный документ представляет собой критический разбор спецификации `WIN10_11_INSTALL_WORKFLOW.md`. В нем выявлены потенциальные уязвимости, архитектурные баги и проблемы автоматизации, которые приведут к сбоям при попытке реализовать инсталлятор "по букве" этого воркфлоу.

---

## 1. Проблема путей и кириллицы (Шаги 1, 3, 5)
**Где в спеке:** Везде используется `$env:USERPROFILE\.context-manager\`.
**Возможные баги и затыки:**
* У многих пользователей Windows имя профиля содержит пробелы (`C:\Users\John Doe`) или кириллицу (`C:\Users\Иван`). 
* Модули Python (особенно `uvicorn`, `onnxruntime`) и `npm` скрипты могут аварийно завершаться или не находить пути, если в них есть нестандартные символы или пробелы без жесткого экранирования.
* **Следствие:** Инсталлятор отработает, но службы `cm-embed` или `cm-api` будут падать при старте.

## 2. Ловушка "Global Python" (Шаг 5.4)
**Где в спеке:** `pip install -r requirements.txt` выполняется напрямую.
**Возможные баги:**
* Это загрязняет глобальную среду Python пользователя.
* В новых версиях Python (с PEP 668) Windows/Linux блокирует `pip install` вне виртуального окружения (ошибка `externally-managed-environment`).
* Возможен конфликт версий пакетов (например, `fastapi` или `onnxruntime`), если у пользователя уже стоят другие версии для его личных проектов.
* **Следствие:** Установка прервется с ошибкой зависимостей. Необходимо использовать `python -m venv .venv`.

## 3. Критический затык базы данных (Шаг 0.5 - 0.7)
**Где в спеке:** "Если 0.7 нет → автоматически создать: `createdb -U postgres context_db`".
**Возможные баги:**
* Команда `createdb -U postgres` потребует пароль. В командной строке PowerShell это вызовет интерактивный запрос пароля.
* Так как скрипт должен быть автоматическим, он "зависнет" в фоновом ожидании ввода пароля. Пользователь увидит просто висящий скрипт, который ничего не делает.
* **Следствие:** Мертвый затык автоматизации. Требуется использование `PGPASSWORD` или предварительно настроенного `pgpass`.

## 4. Баг PowerShell Copy-Item (Шаг 3.1)
**Где в спеке:** "Скопировать исходники проекта... Исключить: node_modules\, dist\, .git\"
**Возможные баги:**
* Команда PowerShell `Copy-Item -Recurse -Exclude` исторически сломана. Она неправильно применяет исключения к вложенным папкам, и в итоге либо скопирует весь мусор (гигабайты `node_modules`), либо упадет с ошибкой слишком длинного пути.
* **Следствие:** Раздувание папки, падение из-за лимитов Windows (260 символов в пути). Нужно использовать `robocopy` или `git clone` локально.

## 5. Ошибки конфигурации NSSM (Шаг 6)
**Где в спеке:** Передача переменных окружения: `nssm set <svc> AppEnvironmentExtra KEY=VALUE`.
**Возможные баги:**
* Для NSSM параметр `AppEnvironmentExtra` ожидает массив строк, разделенных нулевым байтом (`\0`) в реестре. Если в PowerShell вызывать команду `nssm set ... KEY=VALUE` несколько раз подряд, она будет **перезаписывать** предыдущую переменную, а не добавлять новую. В итоге сервис получит только последнюю переменную.
* **Дублирование .env:** Спека говорит считывать `.env` и передавать его в `nssm` для `cm-api`. Это архитектурный костыль. Node.js сам отлично читает `.env` файл. Прошивание переменных в реестр Windows через NSSM приведет к тому, что при изменении `.env` пользователем, настройки не применятся до перерегистрации службы.

## 6. Зависание из-за порядка загрузки (Шаг 7)
**Где в спеке:** `nssm start` запускает службы одну за другой с ожиданием портов (цикл ожидания).
**Возможные баги:**
* После первой перезагрузки компьютера Windows запустит все эти службы **одновременно**. `cm-api` стартанет раньше, чем тяжелая `cm-embed` (которая грузит ONNX модель 10-20 секунд) или `cm-qdrant`. `cm-api` может упасть из-за отсутствия коннекта.
* **Следствие:** Сервисы не работают после ребута ПК. NSSM имеет параметр `DependOnService`, который не указан в спеке, но критически необходим для построения цепочки запуска.

## 7. Проблема Resume и State Machine (Шаг 8)
**Где в спеке:** Чтение и запись в `install.json` массива атомарных операций (`"op":"pip_install", "step":5`).
**Возможные баги:**
* Идея State Machine в bash/powershell очень хрупкая. Если скрипт крашнется (закрыли окно, отключили свет, BSOD) прямо во время записи `Out-File`, `install.json` станет поврежденным.
* Анинсталлер, опирающийся на поврежденный JSON, упадет и оставит в системе мусор.
* Остановка скрипта (Stop) при некритичной ошибке (например, не скачалась какая-то вспомогательная утилита) блокирует весь флоу.

## 8. Неактуальный путь MCP (Шаг 3.6)
**Где в спеке:** Назначение `$env:APPDATA\iflow\mcp\server.js`.
**Возможные баги:**
* Папки `iflow` не существует в стандартах ни Claude Desktop, ни Cursor. Пользователь не сможет подключить MCP, так как инструмент будет спрятан в нестандартной папке.

## 9. Распаковка больших ZIP (Шаг 2.2, 4.2)
**Где в спеке:** Использование `Expand-Archive`.
**Возможные баги:**
* Встроенный в PowerShell `Expand-Archive` крайне медленный на больших файлах (например, на Qdrant или если запаковать модель). При больших I/O операциях скрипт будет висеть минутами, создавая иллюзию зависания, не выдавая прогресс-бара.

## Вывод
Документ `WIN10_11_INSTALL_WORKFLOW.md` действительно содержит архитектурные дефекты, которые делают реализацию прямолинейного и устойчивого инсталлятора по его шагам невозможной. Требуется его предварительный рефакторинг.
