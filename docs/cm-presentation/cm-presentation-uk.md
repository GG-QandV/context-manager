# Context Manager v2.2.1

**Постійна пам'ять для AI-помічників — щоб вони не забували, про що ви говорили**

> Розробник: Євгеній Нам

---

## Що таке Context Manager?

### Проблема

Кожного разу, коли ви говорите з AI, він починає з нуля. Він не пам'ятає, що ви обговорювали вчора, минулого тижня — навіть п'ять хвилин тому. Весь контекст втрачається.

> *Кожна розмова з AI починається "з чистого аркуша" — попередній контекст губиться.*

### Рішення

Context Manager зберігає розмови двома способами: як звичайний текст у базі даних (PostgreSQL), і як "цифрові відбитки сенсу" — числа, які показують, про що саме йшлося.

> *Збереження у двох форматах: текст (для точного пошуку) і "цифровий відбиток змісту" (для пошуку за схожими темами).*

### Результат

AI-помічники пам'ятають минулі розмови, знаходять потрібну інформацію за змістом і можуть обмінюватися контекстом між собою — без участі людини.

> *AI-помічники пам'ятають попередні розмови та можуть обмінюватися інформацією один з одного.*

---

> **Важливе пояснення:** Context Manager використовує стандартний спосіб підключення під назвою **MCP**.
> Уявіть MCP як **універсальний роз'єм (стандарт підключення)** — так само як будь-які навушники з 3.5mm jack підходять до будь-якого телефона, будь-який AI-помічник, який підтримує MCP, може підключитися до Context Manager.
> 
> Якщо ваш AI не підтримує MCP, Context Manager також працює через звичайну веб-адресу (HTTP).

## Архітектура

Система має три логічні шари. Інформація рухається згори донизу та у зворотному напрямку.

![Архітектура Context Manager](img/cm-architecture.svg)

### Як читати цю схему

**Шар 1** — інструменти AI, які ви використовуєте (Antigravity, VS Code тощо).

**Шар 2** — "мозок" Context Manager, який обробляє запити.

**Шар 3** — місце, де живуть дані:

- **PostgreSQL** (реляційна БД — дані в таблицях) — зберігає розмову як текст.
- **ONNX Embedder** — конвертує текст у математичний "відбиток сенсу" (вектор).
- **Qdrant** (векторна БД для пошуку за змістом) — зберігає ці "відбитки сенсу" для швидкого пошуку за змістом.

> **Ключовий момент: обидві бази даних синхронізовані.** Коли ви зберігаєте розмову, текст потрапляє в PostgreSQL, потім ONNX Embedder конвертує його у вектор і зберігає в Qdrant автоматично. Обидві бази завжди містять одні й ті самі дані — просто в різних форматах (текст vs цифровий відбиток сенсу).

## Можливості

Context Manager пропонує кілька інструментів — кожен виконує одну конкретну задачу, ніби кнопки на пульті дистанційного керування.

### Збереження контексту

Три рівні деталізації, залежно від того, скільки ви хочете зберегти:

- **Brief (коротко)** — автоматичний підсумок розмови на 200-300 символів. Ніби твіт про те, що сталося.
- **Important (важливе)** — структуровані нотатки з темами, до 3 000 символів. Ніби підсумок наради.
- **Full (повністю)** — повний лог розмови дослівно. Ніби зберегти весь запис наради.

### Пошук та аналіз

- **Semantic search (пошук за змістом)** — знаходить розмови за сенсом, навіть якщо ви не пам'ятаєте точних слів. Наприклад, пошук "проблеми з сервером" знайде розмови про "помилки підключення до бази даних".
- **Filtered query (фільтр за параметрами)** — пошук з фільтрами: за датою, AI-агентом, режимом збереження.
- **Cross-agent search (пошук між агентами)** — один AI-помічник може знайти те, що записав інший AI-помічник.

### Моніторинг

- **Agent list (список агентів)** — показує всіх AI-помічників та кількість збережених ними записів.
- **Statistics (статистика)** — деталі про конкретного помічника або сесію.
- **Export (експорт)** — вивантаження сесії у JSON (текстовий формат, зрозумілий програмам).

### Вбудовані помічники

- **Content Detector (визначення типу вмісту)** — автоматично розпізнає, що саме читає: код, команди, таблиці чи помилки — і обробляє відповідно.
- **Time Parser (розпізнавання дат)** — розуміє фрази "вчора", "останні 3 дні" українською та англійською.
- **Chunking (розділення тексту на частини)** — розбиває довгі тексти на менші шматки для кращої якості пошуку.

> **НОВИНА!** 4 провайдери LLM-чатів дозволили підключення кастомних MCP: **GPT**, **Claude.ai**, **Perplexity** та **Grok**.

## Підключення та команди

10 інструментів Context Manager для роботи з пам'яттю AI-помічників.

| Команда      | Опис                                                               |
| ------------ | ------------------------------------------------------------------ |
| `cm_save_br` | Зберегти коротко — авто-самарі бесіди/контексту (200-300 символів) |
| `cm_save_im` | Зберегти важливе — структурований запис за темами (до 3K символів) |
| `cm_save_fl` | Зберегти повністю — весь лог сесії + документ дослівно             |
| `cm_search`  | Пошук за змістом — знайти схожі розмови, навіть якщо слова різні   |
| `cm_query`   | Пошук з фільтрами — за датою, агентом, режимом збереження          |
| `cm_cross`   | Пошук у контексті іншого агента — один AI переглядає записи іншого |
| `cm_agents`  | Список усіх AI-агентів з кількістю їхніх записів                   |
| `cm_stats`   | Статистика — скільки записів, коли остання активність              |
| `cm_export`  | Експорт сесії у форматі JSON — вивантажити для аналізу             |
| `cm_help`    | Показати список усіх команд та їхній опис                          |

### Способи підключення

- **Standard MCP (стандартний)** — для AI-програм на вашому комп'ютері. Найпоширеніший спосіб.
- **Web MCP (через інтернет)** — для AI-програм у браузері. Використовує порт `:8770`.
- **Direct HTTP (прямий доступ)** — без MCP, через веб-запити на порт `:3847`. Якщо ваш AI не підтримує MCP.

> **Що таке MCP?** MCP — це єдиний стандарт підключення (мова спілкування) для AI-помічників. Уявіть, що в кожного AI-помічника є "вилка", а в Context Manager — "розетка". MCP — це міжнародний стандарт, якого дотримуються всі виробники, щоб вилка підійшла до розетки.

### PostgreSQL (БД тексту)

24 колонки, 526 записів, 2.9 MB загального розміру. Обладнана розумними індексами для швидкого текстового пошуку.

### Qdrant (БД сенсів)

1,336 векторів, 384 виміри кожен, 13 MB загального розміру. 6 сегментів, статус: справний.

### ONNX Embedder (перетворювач сенсу)

40 MB розмір моделі, INT8-квантування (стиснення для швидкості). ~15 мілісекунд на одне перетворення.

## Технології

Сучасні, легкі технології. Жодних важких фреймворків — працює офлайн, на будь-якому комп'ютері.

### Середовище виконання (Runtime)

| Технологія | Опис                                                     |
| ---------- | -------------------------------------------------------- |
| Node.js    | Середовище для JavaScript — двигун, який запускає сервер |
| TypeScript | Мова програмування — JavaScript із типами для надійності |
| Fastify    | Веб-фреймворк — обробляє HTTP-запити                     |
| Pino       | Журнал подій (логер) — записує, що робить система        |

### Сховища (Storage)

| Технологія    | Опис                                                          |
| ------------- | ------------------------------------------------------------- |
| PostgreSQL    | Реляційна БД — зберігає текст з індексами для пошуку          |
| Qdrant        | Векторна БД — зберігає "відбитки сенсу" для пошуку за змістом |
| ONNX Embedder | ONNX runtime — запускає модель multilingual-e5-small          |

### Інтеграція (Integration)

| Технологія | Опис                                                                    |
| ---------- | ----------------------------------------------------------------------- |
| MCP SDK    | Бібліотека стандартного протоколу — дозволяє AI-помічникам підключатися |
| Axios      | HTTP-клієнт — виконує мережеві запити до інших сервісів                 |
| LangChain  | Використовується лише для розділення тексту — нічого важкого            |

### НЕ використовується

`torch` · `transformers` · `langchain-core` · `redis` · `fastapi` · `hnswlib`

> Цих важких бібліотек свідомо уникали заради швидкості та простоти.

✅ Усі npm-залежності — чистий JavaScript, жодних вбудованих модулів. Працює на Windows, macOS і Linux без змін.

## Запуск та встановлення

Три способи запуску Context Manager, залежно від ваших потреб.

### Docker Compose (3 сервіси)

Docker — це ніби "віртуальна коробка", яка містить усе необхідне для роботи. Автоматичне керування та старт.

| Сервіс          | Порт  | RAM      | Опис                                      |
| --------------- | ----- | -------- | ----------------------------------------- |
| context-manager | :3847 | ~43 MiB  | Головний сервер                           |
| tei-embeddings  | :8080 | ~107 MiB | AI-модель для перетворення тексту на сенс |
| cm-mcp-adapter  | :8770 | ~2.6 MiB | MCP HTTP-адаптер                          |

+ зовнішні: PostgreSQL `:5432`, Qdrant `:6333`

### Нативно (без Docker)

Встановлення безпосередньо на комп'ютер — без віртуалізації.

| Команда         | Дія                                  |
| --------------- | ------------------------------------ |
| `npm install`   | Встановити всі залежності            |
| `npm run build` | Скомпілювати TypeScript у JavaScript |
| `npm start`     | Запустити сервер                     |
| `npm run dev`   | Запуск у режимі розробки             |

> Потрібно: Node.js 18+, PostgreSQL, Qdrant

### Керування процесами

| Платформа | Інструмент               | Опис                                  |
| --------- | ------------------------ | ------------------------------------- |
| Windows   | nssm                     | Запуск як служби Windows (автозапуск) |
| Linux     | systemd                  | Запуск як служби Linux (автозапуск)   |
| macOS     | launchd / Docker Desktop | Запуск як служби macOS                |

### Мережа (Network)

| Компонент            | Опис                                                                     |
| -------------------- | ------------------------------------------------------------------------ |
| orchestrator-network | Спільна Docker-мережа                                                    |
| localhost            | Порти 3847, 6333, 5432, 8080 (тільки локальний комп'ютер)                |
| Docker DNS           | `host.docker.internal` — спеціальна адреса для доступу до хосту з Docker |

## Крос-платформенність

Windows, macOS, Linux — той самий код працює скрізь. Відрізняються лише файли налаштувань.

| Компонент                    | macOS                            | Linux        | Windows                         |
| ---------------------------- | -------------------------------- | ------------ | ------------------------------- |
| npm залежності (бібліотеки)  | ✅ pure JS                        | ✅ pure JS    | ✅ pure JS                       |
| Fastify веб-сервер           | ✅                                | ✅            | ✅                               |
| MCP підключення              | ✅                                | ✅            | ✅                               |
| PostgreSQL клієнт            | ✅ TCP                            | ✅ TCP        | ✅ TCP                           |
| Qdrant клієнт                | ✅ REST                           | ✅ REST       | ✅ REST                          |
| Docker Desktop (опціонально) | ✅ HyperKit                       | ✅ Нативно    | ✅ Не потрібен (рідна установка) |
| ONNX Embedder (AI-модель)    | ✅ CPU                            | ✅            | ✅ CPU                           |
| Шляхи до файлів налаштувань  | `~/Library/Application Support/` | `~/.config/` | `%APPDATA%`                     |

### Підтримка платформ

**Linux** — Повна підтримка. Docker-нативно, systemd-служба, все працює "з коробки".

**macOS** — Повна підтримка. Працює нативно, Docker Desktop, launchd-служба. Шляхи конфігів — `~/Library/Application Support/`.

**Windows** — Повна підтримка. Нативний ONNX runtime, nssm-служба, один-click інсталятор. Все працює "з коробки".
